Perguntas sobre o Git

1 - Para que serve o commit e qual a melhor forma de utilizá-lo?

De forma mais ampla, em ciência da computação, um commit é uma confirmação de alterações feitas, marcando o fim de uma transação.

Em um sistema de versionamento de código, como nosso exemplo o Git, o commit confirma e faz uma marcação para todas as alterações adicionadas à ele, gerando um histórico de commits e sendo possível fazer um checkout em cada um.

Não existe somente uma melhor forma de utilizar o commit, mas existem boas práticas, as quais são muito comuns no mundo do desenvolvimento de software. De tão importante que são as boas práticas que foi criado o Conventional Commits, uma iniciativa que tenta oficializar a forma correta de se fazer commits.

Como boa prática, o commit deve ser feito sempre que um trecho de código for feito e estiver funcionando. Ao fazer esse commit ele deve conter uma mensagem auto explicativa do que foi feito naquela versão. Seguindo o Conventional Commits, essas mensagens devem seguir de um prefixo, que identifique se é uma correção (fix:), nova funcionalidade (feat:) e até correção em produção (hotfix:).

2 - Qual a diferença entre commit e push?

Como visto anteriormente o commit é a confirmação e finalização das alterações adicionadas à ele, já o push “empurra” ou seja faz o envio de todos os commits, ainda não enviados, do repositório local para o repositório remoto.

É possível fazer um push a cada commit realizado, mas também fazer vários commits e depois um push para todos os commits.

Ao fazer o push você mantém o repositório remoto atualizado e em caso de trabalho em equipe, todos poderão baixar a versão mais atualizada do código principal, ou branches paralelas ao projeto.

3 - O que é tag?

As tags no Git funcionam como uma marcação no histórico de commits. Elas são muito utilizadas para determinar versões do código e ou software, por exemplo a versão 1.5.2, seguindo o padrão de versionamento semântico, indica que naquele ponto da história do código, a versão é essa.

Depois de criada uma tag, ela não recebe mais atualizações, ou seja, ela delimita o início e fim dos históricos dos commits realizados. Em caso de novas funcionalidades ou correções, elas deverão ser adicionadas a outra tag. Seguindo o exemplo acima e digamos que foi lançado uma versão de correção, a próxima tag teria a sequência de 1.5.3.

4 - O que é branch?

Uma branch é uma ramificação do código principal, elas são muito utilizadas nos sistemas de controles de versão mais modernos.

Vamos usar o exemplo de um trabalho em equipe, onde precisa ser feito uma correção de uma versão. O desenvolvedor cria uma branch a partir do código principal, trabalha nas correções, faz os devidos commits, faz um push e faz os devidos testes deste código. Depois desse processo e caso esteja tudo certo, essa branch está apta a fazer o merge com o código principal.

Na grande maioria dos trabalhos em equipes, sempre é utilizado branches para o desenvolvimento, seja para correções ou novas funcionalidades.

5 - Como resolver possíveis conflitos no Git?

Os conflitos no Git são muito comuns, principalmente com trabalho em equipe, mas como o Git é uma ferramenta fantástica, ele nos ajuda a resolver esses problemas.

Conflitos de merge ocorrem quando se tem duas ou mais versões do mesmo ponto do código, exemplo, digamos que um desenvolvedor alterou a linha 25 de um arquivo e outro também alterou a mesma linha, ambos fazem o commit e push de suas alterações. Com isso o Git não conseguirá fazer o merge automático dessas alterações pois não sabe qual é a versão que deverá manter, nesses casos é preciso intervenção manual para resolver o conflito.

Após resolver o conflito será preciso fazer um novo commit e push para o repositório remoto. Vale ressaltar que os conflitos de merge ocorrem tanto entre branches quanto entre commits dentro da mesma branch.
